# Remaining Work

The current system can run a full local flow:

1. Upload a PDF.
2. Trigger ingestion (synchronously via the process endpoint, or via the background worker).
3. Run MinerU CLI with `pipeline`.
4. Parse MinerU `*_content_list.json` into page-marked Markdown.
5. Store documents/pages/jobs in PostgreSQL.
6. Upsert/search Qdrant.
7. Return query answers with citations and streamed trace events.

## Recently Completed

- **Background worker**: ingestion can run off the request path through an in-process
  `ThreadPoolExecutor` job queue (`app/services/worker.py`), enabled with
  `USE_BACKGROUND_WORKER=true`. The `submit(job_id)` contract is broker-agnostic and can be
  swapped for Celery/RQ without changing call sites. The synchronous `/process` endpoint remains.
- **Authentication & authorization**: bearer-token login (`/api/auth/login`, `/api/auth/me`),
  HMAC-signed tokens, and per-document ownership isolation. Enabled with `REQUIRE_AUTH=true`;
  users come from `AUTH_USERS` (`username:password:role`). `admin` sees all documents.
- **Document deletion cleanup**: deletion removes the PostgreSQL rows, deletes the Qdrant points
  by `document_id` filter, and removes the stored source/MinerU files. The frontend exposes a
  per-document delete button (with confirmation) that triggers this cascade.
- **Version policy & latest-version UX**: identical content (same hash) is de-duplicated and
  reuses the existing document; a new content hash for the same file name creates an incrementing
  version (`v1`, `v2`, …) and demotes the previous `is_latest`. New endpoints:
  `GET /documents?latest_only=true` and `GET /documents/{id}/versions`.
- **LLM final-answer synthesis**: the final answer is generated by an LLM
  (`app/services/synthesis.py`) with a graceful fallback chain: Agno agent runtime (when
  `USE_AGNO=true`) -> direct LiteLLM call -> extractive fallback.
- **Per-node (PageIndex) embeddings**: ingestion embeds one Qdrant point per TOC node, storing **both
  a dense vector** (node summary embedding) **and a BM25 sparse vector** (payload: `document_id`,
  `node_id`, `heading`, `start_page`, `end_page`, `summary`). The collection (`km_nodes`) is created
  with a named dense vector + a sparse vector using the IDF modifier. A `vector_node` step navigates
  to the most relevant node and reads its precise page range. Deleting a document removes all its node
  points.
- **Precise TOC page ranges**: each node spans from its heading to just before the next heading
  (same page → confined to that page; last node → end of document), instead of defaulting to the
  whole document.
- **Qdrant native hybrid search**: retrieval uses Qdrant's Query API with `prefetch` (dense + sparse)
  and **server-side RRF fusion** (`FusionQuery`) — no in-process fusion. Sparse vectors are built from
  the shared zh/en tokenizer (CJK split to characters) so Chinese matches well, with IDF applied by
  Qdrant. Per-request `strategy` selects `hybrid` | `dense` | `bm25`. The in-process BM25 ranker
  remains only as the offline fallback when Qdrant is disabled.
- **Observability**: OpenTelemetry spans/events export to Phoenix when `PHOENIX_ENDPOINT` is set
  (`app/core/tracing.py`), with optional OpenInference Agno auto-instrumentation. Install the
  exporters/viewer with `uv sync --extra observability`.
- **Frontend document detail view**: a TOC/page explorer with a page-level evidence viewer —
  clicking an answer citation opens the document and highlights/scrolls to the cited pages;
  TOC nodes jump to their page range; version switching is available.
- **Live agent narration (SSE)**: the streamed trace renders as an animated timeline with
  human, confidence-building narration (e.g. "正在翻閱《TXC_SOP_2026.pdf》第 16-17 頁…"), a spinner
  on the active step, and a checkmark on completed steps. Trace events now carry `document_name`.

## Not Yet Production-Complete

- **Deployment packaging**: Docker compose covers PostgreSQL/Qdrant only; backend/frontend/MinerU
  worker containers are still pending.
- **Persistence hardening**: several services keep an in-process cache for the PoC path
  (`USE_DATABASE=false`); production should rely solely on PostgreSQL.
- **Cross-encoder reranker**: the reranker is BM25-based; a model-based cross-encoder would
  improve precision.
